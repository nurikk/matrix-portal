# HD-WF4 Four-Output Driver Design

## Goal

Drive four 64x64 HUB75E panels from a Huidu HD-WF4 with four independent framebuffers and the highest practical refresh rate.

## Hard Constraint

The WF4 is not a four-lane HUB75 controller. The PCB exposes four HUB75E connectors, but reverse-engineered routing shows two RGB data buses and two output banks:

| Connector | RGB bus | Bank gate |
| --- | --- | --- |
| X1 | RGB bus A | GPIO45 high |
| X2 | RGB bus B | GPIO45 high |
| X3 | RGB bus A | GPIO14 high |
| X4 | RGB bus B | GPIO14 high |

`A/B/C/D/E`, `LAT`, `OE`, and `CLK` are shared. This means four independent panels cannot be refreshed simultaneously. The fastest possible stock-PCB design is two-lane bank multiplexing:

1. Enable GPIO45 and scan X1 + X2 with independent RGB data.
2. Enable GPIO14 and scan X3 + X4 with independent RGB data.
3. Repeat continuously.

This gives four independent images, but each bank receives half the scan time.

## ESP32-S3 Peripheral Constraint

ESP32-S3 LCD_CAM can output 16 data bits plus a pixel clock. A two-lane WF4 stream needs:

- 12 RGB bits: 2 HUB75 lanes * `R1/G1/B1/R2/G2/B2`
- `LAT`
- `OE`
- `E`
- Optional spare bit

That exactly fits if `A/B/C/D` are generated by another peripheral and the bank gates are static GPIOs changed only at bank boundaries.

Use RMT for `A/B/C/D`. ESP32-S3 has four TX-capable RMT channels, so this is the natural fit. Put `E` in the LCD data word because there are not enough RMT TX channels for all five address lines.

## Fastest Feasible Output Architecture

### LCD_CAM Data Word

Use a 16-bit LCD_CAM/I80-style DMA stream. Route LCD data bits as:

| LCD bit | WF4 signal |
| --- | --- |
| D0 | bus A R1, GPIO2 |
| D1 | bus A G1, GPIO6 |
| D2 | bus A B1, GPIO10 |
| D3 | bus A R2, GPIO3 |
| D4 | bus A G2, GPIO7 |
| D5 | bus A B2, GPIO11 |
| D6 | bus B R1, GPIO4 |
| D7 | bus B G1, GPIO8 |
| D8 | bus B B1, GPIO12 |
| D9 | bus B R2, GPIO5 |
| D10 | bus B G2, GPIO9 |
| D11 | bus B B2, GPIO13 |
| D12 | LAT, GPIO33 |
| D13 | OE, GPIO35 |
| D14 | E, GPIO21 |
| D15 | unused/debug |

Route LCD PCLK to HUB75 `CLK`, GPIO34.

### RMT Address Timing

Use four RMT TX channels, synchronized with the RMT sync manager:

| RMT channel | WF4 signal |
| --- | --- |
| TX0 | A, GPIO39 |
| TX1 | B, GPIO38 |
| TX2 | C, GPIO37 |
| TX3 | D, GPIO36 |

RMT emits row-address waveforms matching the DMA bank-transfer sequence. Each address state lasts one DMA row-block duration. Consecutive blocks with the same value should be coalesced into longer RMT symbols.

### Bank Gates

Drive bank gates with CPU GPIO only at bank boundaries:

- Bank 0: GPIO45 high, GPIO14 low, LCD stream contains X1 on bus A and X2 on bus B.
- Bank 1: GPIO45 low, GPIO14 high, LCD stream contains X3 on bus A and X4 on bus B.

Bank transfer EOF interrupt toggles gates and starts the next bank transfer. This interrupt rate is low enough: hundreds to a few thousand interrupts per second depending on color depth, not per row.

### Synchronization Guard

LCD_CAM and RMT cannot be started by the same hardware sync manager. Add a short guard region at the start and end of each bank transfer:

- RGB zero
- LAT low
- OE disabled
- E/address set to row 0
- Duration: 8-16 pixel clocks initially, tune with a logic analyzer

Start RMT address streams, then start LCD DMA during the guard. The guard absorbs start skew. At 20 MHz, 16 clocks costs only 0.8 us per bank.

## Framebuffer Model

Expose four independent logical framebuffers:

```c
uint16_t fb[4][64][64]; // RGB565 recommended
```

Recommended initial color path:

- Store app pixels as RGB565 to keep memory low.
- Expand to bitplanes in a render/prep step.
- Use 4-bit or 5-bit output per color initially.

Memory estimate:

| Item | 4-bit | 5-bit | 6-bit |
| --- | ---: | ---: | ---: |
| Four RGB565 framebuffers | 32 KB | 32 KB | 32 KB |
| Two bank bitplane buffers | 32 KB | 40 KB | 48 KB |
| DMA descriptors, packed BCM | ~12 KB | ~21 KB | ~38 KB |

8-bit output is technically possible in memory, but descriptor count and refresh rate make it the wrong target for this board.

## DMA Buffer Layout

Use two bank bitplane buffers:

```c
uint16_t bank_words[2][ROWS_PER_FRAME][COLOR_DEPTH][WIDTH];
```

For 64x64 1/32-scan panels:

- `WIDTH = 64`
- `ROWS_PER_FRAME = 32`
- `COLOR_DEPTH = 4` or `5` initially

For each bank:

- Bank 0 maps `fb[0]` to bus A and `fb[1]` to bus B.
- Bank 1 maps `fb[2]` to bus A and `fb[3]` to bus B.

Each word contains both upper/lower half RGB bits for both buses, `LAT`, `OE`, and `E`.

## Scan Scheduler

Use packed BCM scheduling similar to ESP32-HUB75-MatrixPanel-DMA, not naive full `2^depth - 1` bitplane repetition.

For color depth `d`, packed BCM factor is approximately:

```text
factor = d + (2^(d - 1) - 1)
```

So:

| Color bits/channel | Factor |
| --- | ---: |
| 3 | 6 |
| 4 | 11 |
| 5 | 20 |
| 6 | 37 |
| 8 | 135 |

Bank transfer sequence:

```text
loop forever:
  enable X1/X2 bank
  start RMT ABCD pattern for one bank transfer
  start LCD DMA descriptor chain for bank 0
  wait EOF

  enable X3/X4 bank
  start RMT ABCD pattern for one bank transfer
  start LCD DMA descriptor chain for bank 1
  wait EOF
```

## Expected Refresh Rate

Approximate full-display refresh:

```text
refresh = pixel_clock / (64 * 32 * packed_bcm_factor * 2_banks)
```

For four independent 64x64 panels using two-lane bank multiplexing:

| Pixel clock | 4-bit | 5-bit | 6-bit | 8-bit |
| --- | ---: | ---: | ---: | ---: |
| 8 MHz | 178 Hz | 98 Hz | 53 Hz | 14 Hz |
| 16 MHz | 355 Hz | 195 Hz | 106 Hz | 29 Hz |
| 20 MHz | 444 Hz | 244 Hz | 132 Hz | 36 Hz |
| 22 MHz | 488 Hz | 268 Hz | 145 Hz | 40 Hz |

Practical target:

- Start at 16 MHz, 5-bit output.
- If signal integrity is clean, try 20 MHz, 5-bit.
- Use 4-bit if maximum FPS and low flicker matter more than gradients.
- Avoid 8-bit unless you accept visible flicker.

## Implementation Plan

1. Create a new ESP32-S3 PlatformIO/ESP-IDF target for HD-WF4.
2. Configure flash mode `dio` for the cFeon 8 MB flash.
3. Bring up one-bank two-lane LCD_CAM output with static GPIO address pins first.
4. Add RMT-generated `A/B/C/D` timing and verify with a logic analyzer.
5. Add bank switching between GPIO45 and GPIO14 at LCD DMA EOF.
6. Add four RGB565 framebuffers and bitplane conversion.
7. Add double buffering for app-facing framebuffers only if tearing is visible.
8. Tune pixel clock, latch blanking, OE width, and guard clocks on real panels.

## Validation Checklist

Use a logic analyzer before attaching expensive panels:

- `CLK` frequency matches target.
- `LAT` pulses once per row-block at the intended pixel index.
- `OE` is disabled during address/latch transitions.
- `A/B/C/D/E` select rows 0-31 correctly.
- GPIO45 and GPIO14 are never high at the same time during bank switching.
- X1 and X3 do not ghost into each other.
- X2 and X4 do not ghost into each other.

Then validate with panels:

- Test X1/X2 bank only.
- Test X3/X4 bank only.
- Test all four with unique solid colors.
- Test checkerboard per connector.
- Test row-address pattern for 1/32 scan.
- Measure flicker with camera shutter or photodiode if possible.

## Non-Goals

- True simultaneous four-lane output. The stock WF4 PCB does not expose four RGB buses.
- Compatibility with arbitrary HUB75 scan modes in the first version.
- Wi-Fi during initial bring-up. High-speed HUB75 output can interfere with ESP32-S3 RF; keep it off until display timing is stable.
